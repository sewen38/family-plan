# family-plan-v21-final 更新：发布前必须通过 V21 Release Gate

## 背景

洪雷老师明确指出：虽然定稿模板已经提供，执行策划案输出仍反复和模板不同。根因是此前只做内容递归审核，没有把“模板保真度”作为不可绕过的发布门禁。

## 新增强制门禁

发送任何 V21 执行策划案链接前，必须运行：

```bash
python3 family-plan-pages/scripts/v21_release_gate.py <fusion.html>
```

只有输出 `RESULT: PASS` 才允许发送链接。

若输出 `RESULT: BLOCKED`，必须停止发送，并把阻塞原因如实报告给洪雷老师。

## 执行策划案定稿版格式强制约束

最终输出的执行策划案必须【严格遵循执行策划案定稿版格式】。该要求是发布前硬门槛，不得因赶进度、脚本通过、浏览器能打开或子任务自检通过而降低标准。

### 1. 完整性、可读性、数据来源与问题解决能力

最终执行策划案必须同时满足以下要求：

- 图片、SVG、流程图、架构图、表格和正文内容都必须完整展示，不得缺图、坏图、黑屏、黑块遮字、路径失效或只保留文字描述。
- 全文不得出现乱码、编码异常、符号错乱、文字重叠、文字溢出、表格溢出、手机端横向爆版或打印版不可读。
- 所有文字必须清晰可见：高对比、字号足够、背景不遮挡文字，手机截图也必须能读清核心信息。
- 涉及费用、投资额、官方费、律师费、服务费、生活费、税率、门槛、时间、预算、结佣信息、政策要求等数据，必须有明确数据来源或来源说明。
- 所有政策、法规、税务、身份、资金跨境、教育、福利、居住与合规建议必须符合法规，并能实际解决客户问题；不得用泛化模板、空泛口号或不可执行建议替代。
- 必须保障内容厚度：章节、表格、图示、法案说明、预算、时间轴、风险声明、执行动作不能被摘要化、削薄或用“待确认”空泛替代。

### 2. 客户交付版文本洁净度

输出正文中不能出现任何思考过程语言或内部过程文字，必须是干净的、可直接给客户交付的最终版本。

禁止出现但不限于：

- “我认为 / 我们需要 / 下面生成 / 以下为草稿 / 初审版 / review / clean / generated / TODO / 待人工补充 / 过程说明 / 自检通过 / release gate / quality gate”等内部构建或审核痕迹。
- 文件路径、本地路径、脚本名称、模板调试说明、Agent 子任务过程、模型思考过程、修改记录、提示词痕迹。
- 对客户不可见的内部佣金逻辑除非位于密码保护的内部页，并按定稿模板结构展示。

### 3. 单国家单项目嵌入模块强制标准

单独嵌入的单国家单项目执行策划案必须【严格遵循单国家单项目执行策划案定稿版格式】。

每个单国家单项目嵌入模块必须同时满足：

- 内容维度完整：必须覆盖定稿版要求的国家层、项目层、资金层、身份层、教育层、福利层、税务层、跨境合规层、预算材料层、时间轴流程层、风险声明与法案附件层。
- 内容厚度达标：不得把单项目页压缩成摘要卡片、短表格或几段说明；必须接近定稿样本页厚度。
- 图片和正文内容都必须完整，架构图、流程图、财税图、资金路径图、身份路径图必须可见、清晰、手机端可读。
- 不得出现乱码、文字溢出、表格溢出、图中文字过小、低对比、遮挡、坏图或外链失效。
- 涉及数据必须有数据来源或来源说明；法规、政策、税务、身份路径和执行建议必须符合法规且能实际解决该客户/该项目问题。
- 所有文字必须清晰可见，必须保障内容厚度，不得以“待确认”替代核心数据和核心判断；确实未知时必须说明未知原因、需核验来源和影响范围。

### 4. 同一国家多选项目合并规则

当同一个国家出现多个已选项目时，不得为每个项目机械重复嵌入多个割裂模块，也不得削薄成摘要。

强制规则：

- 同一个国家的所有多选项目必须合并在一个单国家执行策划案嵌入模块中。
- 单国家出现多选几个项目时，使用【单国家执行策划案模版】作为母版，减去未选中项目的相关信息，其余国家层内容完整保留，这就是【单国家多项目执行策划案标准】。
- 必须保留该国家共用的国家层内容：客户信息、根本判断、三分离定位、合规清理、资金归集、财富架构、税务分析、资金跨境、教育规划、福利规划、生活成本、预算框架、材料清单、时间轴、风险声明、法案附件等。
- 必须分别保留每个已选项目的专项内容：申请条件、费用、材料、流程、时间、优劣势、风险、佣金/成本信息、法规来源、身份路径与项目专属执行动作。
- 不得因为多项目合并而删除国家层教育、福利、税务、资金跨境、生活成本、法案附件、架构图或财税图。
- 多项目合并后的模块仍必须满足单国家单项目定稿版格式、内容维度、内容厚度、图文完整性、可读性、数据来源、法规合规和客户问题解决能力要求。

## 诊断草案定稿版模板强制约束

本 skill 同时纳入【诊断草案定稿版-身份财税-V2深度增强】模板作为诊断草案输出的最高优先级格式标准。模板参考文件必须随 skill 一并保存：

- `assets/diagnosis-final-template-v2-deep/index.html`
- `assets/diagnosis-final-template-v2-deep/README.md`

最终输出任何诊断草案时，必须【严格遵循诊断草案定稿版格式】。该要求是硬门槛，不得因赶进度、脚本通过、浏览器能打开、子任务自检通过或内容看似完整而降低标准。

### 1. 诊断草案图文完整、可读、合规、可执行

最终诊断草案必须同时满足以下要求：

- 图片、SVG、流程图、架构图、表格和正文内容都必须完整展示，不得缺图、坏图、黑屏、黑块遮字、路径失效或只保留文字描述。
- 全文不得出现乱码、编码异常、异常替换符、符号错乱、文字重叠、文字溢出、表格溢出、手机端横向爆版或打印版不可读。
- 所有文字必须清晰可见：高对比、字号足够、背景不遮挡文字，手机端截图也必须能读清核心判断、风险、方案和行动项。
- 涉及身份、税务、资金、资产、企业、合规、预算、时间、风险、政策门槛、法案条款等数据和判断，必须有明确数据来源、政策来源、法规来源或来源说明。
- 所有身份、财税、资金跨境、企业合规、家庭资产、风险处置与路径建议必须符合法规，并能实际解决客户问题；不得用泛化模板、空泛口号、不可执行建议或“待确认”堆砌替代。
- 必须保障内容厚度：根本判断、待解决问题、风险分级、重要专题深入分析、多方案对比、推荐路径、行动计划、风险声明、法规附件、材料清单等不能被摘要化、削薄或跳过。
- 诊断草案尤其必须保留定稿版的深度专题结构：重要专题深入分析不得只写结论，必须写清当前风险、为什么会出事、需要核验材料、解决方案、最终交付物。

### 2. 诊断草案客户交付版文本洁净度

输出正文中不能出现任何思考过程语言或内部过程文字，必须是干净的、可直接给客户交付的最终版本。

禁止出现但不限于：

- “我认为 / 我们需要 / 下面生成 / 以下为草稿 / 初审版 / review / clean / generated / TODO / 待人工补充 / 过程说明 / 自检通过 / release gate / quality gate”等内部构建、审核或思考痕迹。
- 文件路径、本地路径、脚本名称、模板调试说明、Agent 子任务过程、模型思考过程、修改记录、提示词痕迹。
- 对客户不可见的内部说明、内部佣金、内部资源调试、内部版本比较，除非定稿模板明确设计为密码保护或内部页。

### 3. 诊断草案模板融入与发布纪律

- 生成诊断草案时，应优先复用 `assets/diagnosis-final-template-v2-deep/index.html` 的商业级 HTML 形态、10章结构、响应式表格、专题深度与法案附件表达方式。
- 不得把诊断草案退化为普通 Markdown、短报告、表格摘要或只有结论的方案。
- 发布前必须进行人工审核：图文完整、无乱码、无溢出、文字清晰、数据有来源、法规可执行、客户交付干净、内容厚度达标。
- 任一项不满足，必须停止发送，继续修复；不得把半成品命名为 final/定稿/最终版。

### 4. 诊断草案定制化生成引擎（2026-07-01 强制升级）

诊断草案的生成必须走 `family-plan-pages/scripts/cloud-runner/diagnosis_template_renderer.py` 的【定制化两阶段生成流水线】，禁止再退化为“一次调用出全部 JSON + 静态兜底”的旧逻辑（旧逻辑会导致每个待解决问题文字雷同、8个专题共用同一段话）。

生成必须满足以下硬规则：

- **每个待解决问题必须是独立定制的专业方案**：基于客户快速问卷的真实数字/国家/账户/天数/身份状态，逐个问题单独调用模型生成，字段必须包含 `law_basis`（真实法条/阈值/罚则）、`lawyer_view`（律师视角）、`tax_view`（财税规划师视角）、`identity_view`（身份规划师视角）、`solution`（分步可执行）、`risk_control`（风险规避）、`action`（立即动作）。禁止不同问题使用相同或雷同文字。
- **专题、方案逐个生成**：8个专题、3个方案均逐条独立生成，每条必须带客户特有的数字/国家/项目/金额/法条，禁止共用话术。
- **法条命中校验**：每个问题的 `law_basis` 必须命中 `references/compliance-tax-law-cn-us-2026.md` 等法条库中的真实条款（如《外汇管理条例》第45条、《刑法》225条非法经营罪500万立案线、IRS SDOP 5%、FBAR/FATCA门槛、37号文、香港保险AML 240万港元、MAS 13O/13U 等）。没命中必须重生成，禁止编造。
- **近似查重（反雷同）**：对问题的 detail/solution/三视角、专题的 current_risk/why_it_happens/solution 做 4-gram Jaccard 近似查重，相似度≥0.6 视为雷同必须重写；不再只拦“完全相同”。
- **多引擎失效转移 + 熔断**：主引擎 `claude-opus-4-8`（`custom-www-primerouter-xyz`，Anthropic 协议），失败自动降级 OpenAI 兼容（gpt-5.5）→ deepseek；网关持续超时时用熔断器停用故障引擎，避免每次都等超时。任一小请求全部引擎失败时，从该问题自身 skeleton 派生（仍客户特定），绝不回退到“8段同文”的死模板。
- **内容厚度参照定稿版**：最终 HTML 必须达到或超过定稿版模板厚度（≥12章、≥13表、≥18KB），并通过 `diagnosis_data_is_adequate()` 校验（≥3问题各带三视角+法条、≥8专题五段式、≥3方案、≥3法案）。
- **两级问卷两级产物**：快速问卷 → 定制诊断草案；快速问卷 + 评估问卷 → 定制执行策划案。诊断草案的问题/专题/方案必须由客户填写内容驱动，不得套通用模板。
- **手机端可读**：定稿模板已加移动端媒体查询（hero 标题换行、副标题高对比、表格 overflow-x 滚动），发布前必须实测手机端标题不裁切、副标题清晰、表格可横向滚动。

环境变量：`ANTHROPIC_API_KEY`/`ANTHROPIC_BASE_URL`/`ANTHROPIC_MODEL` 配主引擎；`OPENAI_API_KEY`/`OPENAI_BASE_URL`/`OPENAI_MODEL` 配降级引擎；`OPUS_BREAKER_THRESHOLD`（默认2）控制熔断阈值；`DIAG_MAX_PROBLEMS`（默认6）控制问题数上限以约束运行时长。

## Release Gate 必检项

1. 模板注册表存在且最高优先级模板可读取。
2. 原始 zip SHA256 与注册表一致。
3. pristine 解压目录与原始 zip 字节一致，证明模板未被污染。
4. 融合页必须保留 pristine 模板结构：
   - 快速目录
   - 数据完整性检查表
   - 完整单项目模块嵌入区
   - 15章大框架
5. 模板块顺序必须为：
   - 快速目录 → 数据完整性检查表 → 完整单项目模块嵌入区 → 第1章
6. 禁止自造替代模板模块，例如：
   - 页面交付质量摘要
   - Quality Gate
   - Project Modules
   - 11个完整项目模块清单
7. 客户可见文本不得暴露内部构建痕迹、文件路径、generated/clean/review/manual/template路径等。
8. 必须递归运行所有单项目页审核；任一单项目页 BLOCKED，总体 BLOCKED。
9. 必须检查最终执行策划案是否严格遵循执行策划案定稿版格式，并满足图文完整、无乱码、无文字溢出、有数据来源、符合法规、实际解决问题、文字清晰可见、内容厚度达标。
10. 必须检查客户可见正文是否为干净交付版，不得出现思考过程语言、内部过程说明或模板调试痕迹。
11. 必须检查所有单国家单项目嵌入模块是否严格遵循单国家单项目执行策划案定稿版格式，且内容维度、内容厚度、图文完整性、可读性、数据来源、法规合规和解决问题能力均达标。
12. 必须检查同一国家多选项目是否已合并为一个单国家执行策划案嵌入模块，并按“单国家执行策划案模版减去未选中项目信息，其余保留”的标准执行。
13. 输出诊断草案时，必须检查是否严格遵循诊断草案定稿版格式，并满足图文完整、无乱码、无文字溢出、有数据来源、符合法规、实际解决问题、文字清晰可见、内容厚度达标。
14. 必须检查诊断草案客户可见正文是否为干净交付版，不得出现思考过程语言、内部过程说明或模板调试痕迹。

## 执行纪律

- 脚本初筛 PASS 不等于 release gate PASS。
- 递归内容审核 PASS 不等于 release gate PASS。
- 浏览器能打开不等于 release gate PASS。
- 子任务自检 PASS 不等于父级人工审核 PASS。
- 只有 release gate PASS + 浏览器线上验证 + 最终审核报告存在 + 按人工审核标准确认图文完整/无乱码/无溢出/有来源/法规可执行/客户交付干净/内容厚度达标，才允许发执行策划案链接。
- 诊断草案必须通过诊断草案定稿版人工审核标准：严格遵循诊断草案定稿版格式，图文完整、无乱码、无溢出、有数据来源、法规可执行、实际解决客户问题、客户交付干净、内容厚度达标，才允许发链接。
- 如果任一项不满足，必须停止发送，继续修复；不得把半成品命名为 final/定稿/最终版。
